The Trailhead Brief

A social, Batch-style outdoor trip planner built entirely on free public data (NPS, Recreation.gov/RIDB, OSM/USGS) — with permits and campsite availability live in the app, trip-building done on desktop, and a wedge into college students instead of going head-to-head with AllTrails on day one. This doc captures everything dug up so far so nothing has to be re-researched.

00 — The Concept

Trail Planning + Permits + Vibes, in One App

Combine trail planning and campsite/permit reservation through the Recreation.gov API — this gets you live trail data and live availability, just like Gaia. Tiles cost a little money to load and take a moment to render properly, but it's feasible to match the core feature set of AllTrails and Gaia without their budgets.

PILLAR 1
Front end = Batch energy
User-facing app is hooked, grabby, Instagram-style — reels, vibe checks, chat/poll-the-group mechanics. Mobile-first for discovery and social.
PILLAR 2
Planning = desktop-first
The actual "create a trip" flow lives on Mac, not phone — a deliberate bet against mobile-only trends, modeled on AllTrails' simplicity.
PILLAR 3
Data = Rec.gov API
Permit + campsite availability calculated from the Recreation.gov API — not necessarily full topo/tile rendering. Permit availability is live and already widely accepted as a category.
PILLAR 4
Wedge = college students
Desktop planning is a feature, not a bug, for this audience — students are on laptops in class far more acceptably than on their phones. Niche down before going broad, the way Batch wedged into bachelorette parties first.

The reference point for the planning-flow UI is the interactive John Muir Trail guide below — simple, legible, resupply-point-labeled, mileage between every stop. The instinct: nobody else makes it this easy to follow, and they should.

The design north star for the desktop trip-builder flow — clean point-to-point mileage, labeled resupply points, no unnecessary topo clutter. Source: outdoorstatus.com/articles/john-muir-trail-interactive-guide. The standing question from this thread: why does every competitor make this so much more complicated than it needs to be?

Betting against the market trend of "mobile-heavy" by design: planning happens on Mac because it's easier to visualize complex multi-day trips, and because the target market (college students) finds laptop use in class far more normalized than phone use. Treated here as a tactical choice, not an oversight.

02 — The Comp Company

Batch (The Bach LLC): Funding, Model, Investors

Founded 2018, product launched March 2020. Originally branded "Bach" (bachelorette-party planning), later rebranded to "Batch" while expanding into all party types. Legal entity on the App Store is listed as The Bach LLC.

Revenue model: a subscription "billboard" model — local vendors (boat captains, hibachi chefs, party-bus operators) pay Batch a flat subscription fee for marketplace visibility rather than Batch taking a commission per booking. 03 — The Public Data Layer

What's Actually Free to Build On

NPS App & NPS Data API

The official NPS App is a federal government product — no investors, no valuation, funded by congressional appropriation. Development and ongoing maintenance were supported by a $1 million congressional appropriation (the only hard dollar figure found tied specifically to the app), with production work running through Harpers Ferry Center, NPS's internal media division.

U.S. federal government works are public domain by default (17 U.S.C. §105) — reuse the data freely. The one hard exception: the NPS Arrowhead logo is a protected trademark (Lanham Act + 18 U.S.C. §701) — never use it to imply official endorsement.

The official, sanctioned developer channel: developer.nps.gov — free API key, covers alerts, articles, campgrounds, events, lesson plans, news releases, parks (with photos), and visitor centers.

Recreation.gov / RIDB

The bigger one — covers 12 federal agencies (NPS, Forest Service, BLM, Army Corps of Engineers, Fish & Wildlife, Bureau of Reclamation, etc.) under one database. Free, Creative Commons Attribution licensed, explicitly opened for third-party innovation, no need to ask permission first. Base endpoint: https://ridb.recreation.gov/api/v1/. Rate limit: 50 requests/minute.

OpenStreetMap & USGS

Gaia GPS's own primary map data is this free data — their base "Gaia Topo" layer is OSM-based, styled to resemble USGS topo quads. Two licensing notes that matter: (1) OSM is ODbL-licensed — a rendered map only requires attribution; share-alike only triggers if you redistribute the raw combined database itself. (2) Map styling (colors, line weights, icon sets, shading technique) is separately copyrightable as creative expression, even though the underlying geographic facts aren't — build on the same data, just don't visually clone Gaia Topo's specific look.

State Parks (e.g. California)

Different legal regime — the federal public-domain rule does not extend to states. California State Parks data is free to use but explicitly copyrighted: attribution required, no resale, no alteration of the dataset. No 50-state unified API exists; coverage and quality vary wildly state to state.

Where this idea is genuinely strong

  • The data foundation is real, not hypothetical — RIDB, NPS, OSM, USGS all check out as legitimately free and reusable.
  • "Social + grabby" is an actual gap — nobody has done the Batch reframe (logistics problem → vibe people want to post about) for outdoor trips yet.
  • Refer-out, not transact — sidesteps the hardest regulated part (processing federal/state payments) while still giving users a reason to stay in-app.

Where this gets hard — the honest version

  • Not a vacuum. AllTrails has 90M+ users and PE backing (Permira/Spectrum, ~$150M) and is moving fast on social. The real risk is attention and incumbency, not data or engineering.
  • Social network effects are brutal to bootstrap solo — AllTrails took over a decade; Batch took real venture money even within a narrow niche before broadening.
  • Data quality becomes your problem — RIDB itself warns federal agencies are responsible for quality and that data may be missing/incomplete.
  • Desktop-for-planning is a contrarian bet — defensible for the college-student wedge specifically, but narrower than "general public facing."
05 — The Technical Proof

Permit Availability Is Already Live (Case Study: Kearsarge Pass)

Tested directly against Inyo National Forest's wilderness permit system (covers Kearsarge Pass and dozens of other Eastern Sierra trailheads, recreation.gov facility ID 233262). Found the exact mechanism recreation.gov's own frontend uses, pulled from working open-source code:

Step 1 — raw availability, no API key required (this is recreation.gov's own public internal endpoint):

https://www.recreation.gov/api/permitinyo/233262/availability?start_date=2026-06-01&end_date=2026-06-30&commercial_acct=false

Returns JSON for every Inyo NF trailhead at once, organized by date → entrance code → remaining slots.

Step 2 — resolve a trailhead name to its entrance code (needs a free RIDB key):

https://ridb.recreation.gov/api/v1/facilities/233262/permitentrances?query=Kearsarge header: apikey: YOUR_KEY

This confirms the data genuinely is there and genuinely is reachable without scraping the rendered page. Open item: never got to actually pull live numbers in-session — blocked by sandbox network rules (recreation.gov isn't on the allowed domain list) and no Chrome browser was connected for live navigation. The fix is trivial outside this environment: register a free RIDB key and run the two calls above directly.
